# Signing & push notifications — how to sign under your own Apple account

How to build a real, self-signed `.ipa` from this fork so that **push
notifications and other entitlement-gated features work**, and how to
distribute it to a small group (friends / testers) without the App Store.

> TL;DR: the whole app identity is derived from **two values**
> (`bundle_id` + `team_id`) in one config JSON. Change those, provide your
> own signing certificate + provisioning profiles (one per extension), and
> the build templates every bundle id, App Group, and entitlement for you.
> Re-signing a prebuilt `.ipa` afterwards (Sideloadly etc.) **breaks
> push** — sign at build time instead.

---

## 1. Why re-signing breaks notifications

The current CI (`.github/workflows/build.yml`) produces an **unsigned**
build:

- line 65: `ImportCertificates.py --path build-system/fake-codesigning/certs` — **fake** certs
- line 70: `--configurationPath="build-system/appstore-configuration.json"` — the **official Telegram** identity (`ph.telegra.Telegraph`, team `C67CF9S4VU`), not yours
- line 71: `--codesigningInformationPath=build-system/fake-codesigning` — **fake** profiles

That `.ipa` is then re-signed off-device (Sideloadly on Windows). Re-signing
rewrites the bundle id and drops the App Group entitlement, so the
Notification Service Extension can no longer read the shared container and
**cannot decrypt push payloads**. You get either no pushes or empty
"You have a new message" notifications.

To fix this, signing must move **into** the Bazel build with your own
identity. Once that is done, Sideloadly is no longer needed.

---

## 2. Single point of control

App identity is NOT hard-coded in `BUILD` files. It flows from a config JSON
→ generated `variables.bzl` → everything else.

- `build-system/Make/BuildConfiguration.py` reads the JSON and writes
  `variables.bzl` (`telegram_bundle_id` line 46, `telegram_team_id` line 49,
  `telegram_aps_environment` — auto-resolved from the profile, line ~209).
- Reference output: `build-system/example-configuration/variables.bzl`
  (`telegram_bundle_id = "ph.telegra.Telegraph"`, `telegram_team_id = "C67CF9S4VU"`).
- Consumed in `Telegram/BUILD` (loaded lines 48–54).

From `bundle_id` + `team_id` alone, the build derives:

| Thing | Where | Value |
|---|---|---|
| Main app id | `Telegram/BUILD:1984` | `{bundle_id}` |
| NSE id | `Telegram/BUILD:1763` | `{bundle_id}.NotificationService` |
| Share / Widget / SiriIntents / NotificationContent / BroadcastUpload | `Telegram/BUILD:1118 / 1329 / 1466 / 1224 / 1702` | `{bundle_id}.<Suffix>` |
| App Group | `Telegram/BUILD:552` | `group.{bundle_id}` |
| `application-identifier` | `Telegram/BUILD:555` | `{team_id}.{bundle_id}` |
| `aps-environment` | `Telegram/BUILD:544` | from config / profile |

The App Group is derived identically at **runtime**, so code and entitlements
stay in sync automatically:

- `Swiftgram/SGAppGroupIdentifier/Sources/SGAppGroupIdentifier.swift:24` — `group.\(sgBaseBundleIdentifier())`
- `submodules/TelegramUI/Sources/AppDelegate.swift:301-305` — `appGroupName = "group.\(baseAppBundleId)"`

There are **no checked-in `.entitlements` files** — the main-app entitlements
are generated by the `TelegramEntitlements` `plist_fragment`
(`Telegram/BUILD:587-604`). Extensions get their entitlements from their
**embedded provisioning profiles**, not from a BUILD `entitlements =`
attribute — this is why every extension needs its own profile (see §4).

## 3. Step 1 — your config JSON

Copy `build-system/template_minimal_development_configuration.json` and fill
in your values:

```json
{
  "bundle_id": "org.yourname.Swiftgram",
  "api_id":  "<from https://my.telegram.org/apps>",
  "api_hash": "<from https://my.telegram.org/apps>",
  "team_id": "<your 10-char Team ID>",
  "app_center_id": "0",
  "is_internal_build": "true",
  "is_appstore_build": "false",
  "appstore_id": "0",
  "app_specific_url_scheme": "tg",
  "premium_iap_product_id": "",
  "enable_siri": false,
  "enable_icloud": false,
  "sg_config": ""
}
```

Find your **Team ID** in Keychain Access → your Apple Development/Distribution
certificate → *Get Info* → *Organizational Unit* (also shown on
developer.apple.com → Membership).

---

## 4. Step 2 — Apple Developer setup (App IDs, capabilities, profiles)

Requires a **paid** Apple Developer membership ($99/yr). On
developer.apple.com:

1. **App Group** — Identifiers → App Groups → create `group.{bundle_id}`.
2. **App IDs** — create one per target below. Enable **App Groups** on all of
   them (attach the group above); enable **Push Notifications** on the main
   app id.

   | App ID | Push | App Group |
   |---|---|---|
   | `{bundle_id}` (main app) | ✅ | ✅ |
   | `{bundle_id}.NotificationService` (**required for push**) | — | ✅ |
   | `{bundle_id}.Share` | — | ✅ |
   | `{bundle_id}.Widget` | — | ✅ |
   | `{bundle_id}.SiriIntents` | — | ✅ |
   | `{bundle_id}.NotificationContent` | — | ✅ |
   | `{bundle_id}.BroadcastUpload` | — | ✅ |

3. **Provisioning profiles** — one per App ID above, of the type matching how
   you distribute (see §6). For friends → **Ad Hoc**, with their device UDIDs
   registered under Devices.
4. **(Optional) APNs Auth Key** — Keys → create an `.p8` key with Apple Push
   Notifications enabled, if your setup needs a server-side key. Telegram's own
   servers deliver the encrypted push that the NSE decrypts locally; the
   critical piece on the client is the correctly-signed NSE + App Group.

## 5. Step 3 — wire your signing into the build

Create a codesigning directory modeled on `build-system/fake-codesigning/`:

```
your-codesigning/
├── certs/                                  # your .p12 (Distribution or Development)
└── profiles/
    ├── {bundle_id}.mobileprovision
    ├── {bundle_id}.NotificationService.mobileprovision
    ├── {bundle_id}.Share.mobileprovision
    ├── {bundle_id}.Widget.mobileprovision
    ├── {bundle_id}.SiriIntents.mobileprovision
    ├── {bundle_id}.NotificationContent.mobileprovision
    └── {bundle_id}.BroadcastUpload.mobileprovision
```

`BuildConfiguration.py` matches profiles by the `team_id.bundle_id` prefix
(lines ~201-202, 229-230), so profile filenames/contents must match your ids.

Then point the build at your config + signing (in `build.yml` or a local
`Make.py` invocation):

```bash
python3 build-system/Make/ImportCertificates.py --path your-codesigning/certs
python3 -u build-system/Make/Make.py \
  --bazelUserRoot="$BAZEL_USER_ROOT" \
  --overrideXcodeVersion \
  build \
  --configurationPath="build-system/my-configuration.json" \   # was appstore-configuration.json
  --codesigningInformationPath=your-codesigning \             # was fake-codesigning
  --configuration=release_arm64 \
  --buildNumber="$BUILD_NUMBER"
```

The resulting `.ipa` is signed with your certificate and carries working
`aps-environment` + `group.{bundle_id}` entitlements. **Do not re-sign it.**

> Secrets: never commit real `.p12`/`.mobileprovision`. In CI, store them as
> base64 GitHub Secrets and unpack into `your-codesigning/` at the start of
> the job, or keep them in a private repo (`GitCodesigningSource` in
> `BuildConfiguration.py:268` supports this; branch = team_id).

## 6. Distribution: who can install the result

The **provisioning profile type** decides who can install the signed `.ipa`.

| Profile type | Who can install | Push works | Use for |
|---|---|---|---|
| Development | only devices registered in the profile (by UDID) | ✅ | yourself + a few test devices |
| **Ad Hoc** | up to **100 devices/type per year**, by UDID | ✅ | **friends / testers** |
| App Store | anyone, via App Store | ✅ | not applicable here |
| Enterprise ($299/yr) | anyone, by link | ✅ | needs D-U-N-S + org; against ToS for public sharing |

There is **no** legal way to "sign once, install anywhere" without Enterprise.

### Ad Hoc notes for distributing to friends

- Collect each friend's **UDID**, register it under Devices, regenerate the
  profiles, then build **one** `.ipa` containing all their UDIDs.
- Adding a **new** friend later → regenerate profile + rebuild; **only the new
  person** reinstalls. Existing installs keep working — the added UDID extends
  the allow-list, it does not invalidate prior signatures.
- The signing **certificate/profiles expire (~1 year)** → at that point
  everyone must reinstall a freshly-signed build. This annual reinstall is the
  main reason to use a paid account: a free Apple ID drops that to **7 days**.
